# 机械手表生成器等价迁移规范

**状态：** 待实施  
**日期：** 2026-07-11  
**适用范围：** 从 `text-to-cad` 提取机械手表生成器为独立开源项目，并保留未来回流上游的能力。

## 1. 目标

建立一个独立、可开源、可验证的机械手表生成器项目。其第一版必须与 `text-to-cad` 中已经用户验收的生成器行为等价，并包含三种正式布局：

1. **模式 1：中心显示。** 时针、分针同轴居中，秒针偏心。
2. **模式 2：分离显示串联传动。** 分针、时针独立轴位，动力经串联显示机构传递。
3. **模式 3：分离显示独立传动。** 分针与时针各自拥有独立显示动力支路。原模式 4 升格为此模式；旧模式 3 不纳入正式发布。

每个模式都交付完整动力链、轮系、轴系、宝石轴承、三针显示、桥板、服务支撑岛、沉头螺钉、外部 Swiss lever 擒纵机构、材料外观、透明度、运动预览、语义 sidecar 与工程验证。

## 2. 非目标

- 本迁移不重新设计机械表拓扑，不改变已验收的视觉和工程规则。
- 本迁移不以“更干净的架构”为理由简化几何、材质、运动、语义或验证。
- 本迁移不承诺生产级钟表制造设计。
- 本迁移不修改 `text-to-cad` 的既有实现、提交历史或验收模型。
- 本迁移不把第三方擒纵机构资产伪装成项目自研资产。

## 3. 总原则

### 3.1 等价优先于重构

先完整复制每个模式的已验收生成闭环，再考虑抽取共享模块。任何新抽象只能在新旧工件逐项一致后引入。

### 3.2 单一设计真源

每次生成必须形成一个冻结的 `DesignIR`。以下所有工件都只能从这同一对象派生：

- STEP 几何与零件 occurrence 清单；
- 求解报告、请求种子与实际种子；
- occurrence 级语义 sidecar；
- 角色合同；
- 材质、颜色和透明度合同；
- 运动学与动画绑定；
- 6DOF、啮合、包络、干涉、桥板和可制造性验证报告。

禁止由一个新 solver 写 JSON、另一个 legacy backend 重新求解并画 STEP 的双真源结构。

### 3.3 失败必须显式

无可行解、需要替换 seed、外部资产缺失、材质未写入、motion 未绑定、语义未覆盖、验证失败都必须返回失败状态和结构化原因。禁止静默改 seed、忽略失败或通过放宽 validator 放行。

### 3.4 工件可审计

每个发布模型都要能回答：每个可见零件是什么、承担什么角色、由何种设计变量决定、安装在哪个局部坐标系、如何运动、采用何种材质、由哪些验证规则放行。

## 4. 上游可回流约束

未来若与 `earthtojake/text-to-cad` 作者达成合作，项目应能低摩擦地合并为上游的一个机械设计领域扩展。为此：

1. **上游工具链不 fork。** 使用 build123d、OpenCascade、CAD Explorer 和 text-to-cad 的公开接口；不得修改上游核心以满足本项目需求。
2. **领域代码独立。** 手表逻辑放入单一领域包，不能散落在 CLI、viewer 或上游适配代码中。
3. **适配器很薄。** 唯一的 text-to-cad 适配层负责 STEP/GLB/Explorer 调用和工件发布，不参与布局、几何、语义或验证决策。
4. **可移植的依赖方向。** `watch_domain` 只能依赖标准 Python、build123d/OpenCascade 和稳定的项目协议；不得反向依赖新项目的 CLI 或目录绝对路径。
5. **贡献形态可读。** 每个未来可回流模块必须有清晰的公共入口、测试、许可证说明和最小依赖清单。
6. **无私有运行时绑死。** 不把本地用户路径、私有服务、LLM API key 或临时浏览器端口写入领域逻辑。

## 6. 每个模式的迁移闭包

迁移单个模式时，必须一起复制并锁定下列内容：

1. 拓扑定义、变量域、随机种子处理与可行性求解；
2. XY、Z-stack、齿轮啮合相位、外壳包络和桥板服务带规则；
3. 全部几何构造函数、布尔切除、桥板分割与镂空；
4. 外部擒纵机构的完整 leaf assembly、变换、替换件、来源映射；
5. occurrence 级零件图谱、局部坐标系、装配关系与 6DOF 约束；
6. 材质分类、RGBA、透明度、STEP/GLB 叶子实体颜色同步；
7. 动画目标、旋转轴、方向、速比、采样帧验证和 Explorer sidecar；
8. 角色合同、语义 sidecar、求解报告、运动学报告、验证报告；
9. 失败案例、固定基线 seed 和视觉基线截图。

任何一项缺失，该模式只能标为“迁移中”，不能进入发布矩阵。

## 7. 发布工件合同

每次成功生成至少发布：

```text
model.step
model.step.glb
model.step.js
design-ir.json
solver.json
semantic.json
role-contracts.json
placement-manifest.json
motion.json
kinematic.json
validation.json
pattern-manifest.json
MANIFEST.json
```

`MANIFEST.json` 对所有工件计算哈希。`validate` 必须重放工件完整性检查，并验证 sidecar occurrence 集合与最终 STEP occurrence 集合双向相等。

### 7.1 基线捕获的正式编排链

原项目的三个模式均由多个正式阶段共同交付：基础求解与工程证据、最终桥板或完整装配、STEP 到 GLB 的转换，以及浏览器运动/透明度同步。因此，迁移基线不得把任一中间建模函数误判为完整生成器。

基线捕获必须在**同一临时运行目录**中依次调用每个模式既有的正式阶段，并记录每一阶段的入口、输入、输出和失败状态：

1. **模式 1：** 基础工程证据阶段 -> 最终分析桥板/镂空/外部擒纵完整装配阶段 -> STEP/GLB 转换 -> 浏览器工件同步。
2. **模式 2：** 分离显示基础工程证据阶段 -> 分区桥板/镂空完整装配阶段 -> Pattern 2 桥板复核阶段 -> STEP/GLB 转换 -> 浏览器工件同步。
3. **模式 3：** 原模式 4 的完整独立显示硬门禁装配阶段 -> STEP/GLB 转换 -> 浏览器工件同步；其报告中已有的源生证据可被逐字节提取为规范 sidecar，但不得重新推断或重新求解。

每个阶段产生的源生证据分为三类：

- **源生工程证据：** 原函数直接生成的求解、语义、角色、运动学、验证、桥板或完整报告；迁移工具只能原样收集或从同一报告中无损提取。
- **确定性派生发布工件：** GLB、截图、哈希清单和发布清单；允许在最终 STEP 后由既有工具链确定性生成，并必须标注派生来源。
- **规范化聚合工件：** 为公开 API 统一命名而创建的 `design-ir.json`、`semantic.json` 等；只能引用或无损汇总已验证的源生证据，禁止从 STEP 反推语义或用第二个求解器补写内容。

多阶段工件被允许进入同一发布包的前提是通过**同源设计指纹**校验。指纹至少包含：请求/实际 seed、模式标识、选中候选标识、所有轴位、齿轮齿数/模数/节圆或齿顶半径、桥板布局标识，以及最终 STEP 名称。任一阶段指纹不相等、证据缺失或发生静默 seed 替换，整次基线捕获必须失败。

## 8. 不可放宽的验收门禁

### 8.1 设计和几何一致性

- `DesignIR` 记录的每个齿轮、轴、桥板、螺钉、轴承和外部 leaf 都存在于实际 STEP；
- STEP 中无未声明的可见零件；
- 求解报告、语义 sidecar、角色合同和 placement manifest 都引用同一 `design_id` 与同一实际 seed；
- requested seed 与 resolved seed 必须相同。若策略允许搜索候选 seed，输出必须显式标识候选集合和选中原因，且不能冒充原 seed。

### 8.2 运动与装配

- 所有运动 occurrence 有唯一局部旋转轴和父级安装链；
- 固定件没有平移或公转自由度；
- 齿轮、指针和主发条盒动画只围绕设计轴旋转；
- 秒针、分针、时针的角速度比例与方向满足该模式的动力学合同；
- 6DOF、扫掠、轴避让、指针包络和支承覆盖均以最终 occurrence 几何验证。

### 8.3 几何与制造规则

- 每对啮合齿轮的齿数、模数、中心距、相位均从最终设计对象导出并验证；
- 齿顶包络到壳体内壁满足安全余量；
- 桥板接缝宽度、服务岛、螺钉端部布置、轴承覆盖、最小腹板、镂空连续曲线和布尔窗口均在最终实体上复核；
- 材质与透明度逐 occurrence 写入并在 GLB 中验证。

### 8.4 人工视觉门禁

每个模式的基线 seed 至少提供俯视、等轴和动画预览。自动检查全部通过后，仍需执行一次 agent 自检，才可提交给用户目视验收。

## 9. 等价迁移验收矩阵

每个模式选取原项目中已通过验收的固定 seed，并以其正式编排链生成原项目基线，再同时跑新旧版本。对每个基线记录：

- 工件清单和哈希；
- occurrence ID 集合；
- 角色、材质和透明度；
- 轴位、齿轮参数、桥板边界与螺钉位置；
- 运动 target、旋转轴、速比与方向；
- 验证检查项、失败原因与最终状态；
- 俯视和等轴截图。

基线 seed 固定为：模式 1 `731`，模式 2 `55330`，模式 3 `731`。这些 seed 与对应的正式完整生成链绑定；不得以某个中间阶段的示例 seed 替代。

允许 STEP 逐字节不同，但只允许经过批准的几何容差差异。任何零件丢失、额外零件、角色丢失、颜色丢失、运动失效、验证项减少或 seed 漂移都视为迁移失败。

## 10. 实施顺序

1. 建立空的新仓库、许可证与来源目录，不迁入任何生成代码。
2. 从原仓库的正式编排链冻结每个模式的基线工件和验收清单，并验证多阶段设计指纹一致。
3. 完整迁移模式 1，并通过全量等价矩阵后冻结。
4. 完整迁移模式 2，并通过全量等价矩阵后冻结。
5. 完整迁移模式 3，并通过全量等价矩阵后冻结。
6. 迁入共享工件协议、材料/运动/语义验证；仅接受从已冻结模式反向抽出的公共模块。
7. 进行多 seed 回归、负例测试、clean-clone 安装、人工视觉验收与开源发布审查。
8. 只有三种模式均通过后，才讨论将目录和接口整理为可回流 text-to-cad 的贡献包。

## 11. 停止与回退规则

- 任一模式的 parity 失败：停止抽象化，回到该模式的完整基线实现定位差异。
- 出现双真源、静默 seed 替换、validator 放宽或工件缺失：立即阻断发布。
- 无法确认第三方资产许可：保留生成接口和 mock 验证，默认不将资产打包发布。
- 不删除或改写 `text-to-cad` 中的基线模型；所有试验输出只写入新仓库外的临时目录。

## 12. Pattern 3 完整证据基线

经明确批准，原 `text-to-cad` 的 Pattern 4 正式完整入口已在一次最小、可测试的源端修改后，持久化该次运行的四类原始 payload：求解、语义、角色合同与运动学。它们写入同一份 complete report，并附带完整入口卡片标识、payload 原始卡片标识、实际种子与构造函数来源。该改动不改变几何构造、硬门禁、STEP 导出或模型运动行为。

- Pattern 3（原 Pattern 4）与 Pattern 1、Pattern 2 一样进入完整等价迁移和认证矩阵。
- 每次 Pattern 3 基线捕获必须来自包含四类原始 payload 的同一次正式 complete report；禁止从 STEP 反推、跨运行拼接或重新求解来补齐证据。
- 完整入口源端修改固定为提交 `5be78528`，迁移捕获以该提交的归档内容运行；该提交之后源工作树重新按只读处理。
- 四类 payload 必须保留原始内容。种子、入口与来源信息只存在于外层 provenance 包装，不得改写 payload。

## 13. 成功定义

新项目在干净环境中可以为模式 1、2、3 的固定基线和多 seed 矩阵生成完整模型；每个模型的 STEP、语义、角色、材质、运动和验证报告引用同一 `DesignIR`；所有硬门禁与人工视觉审查通过；领域包可在不改动上游核心的前提下作为独立贡献模块接入 `text-to-cad`。
